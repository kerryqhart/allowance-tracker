Fix calendar date selection: Replace string dates with DateTime objects

**MAJOR ARCHITECTURE FIX**: Resolves transactions being created on wrong calendar day

**Root Cause**: Frontend was passing "YYYY-MM-DD" strings but backend expected 
RFC 3339 format, causing "Invalid date format: premature end of input" errors.

**Solution**: Implement proper DTO architecture with DateTime objects throughout:

Frontend Changes:
- Create DateTime<FixedOffset> objects from calendar selection (noon Eastern Time)
- Remove string date parsing - construct valid DateTime objects immediately

DTO & Backend Changes:
- Update all request DTOs to use DateTime<FixedOffset> instead of Option<String>
- Simplify transaction service - remove complex string parsing fallback logic
- Update validation methods to accept DateTime references
- Maintain "string dates never leave CSV layer" architectural principle

Files Modified:
- shared/src/lib.rs: DTO structures with DateTime fields
- egui-frontend/src/ui/app_state.rs: Frontend DateTime object creation
- backend/domain/: Simplified validation and transaction creation logic

**Result**: Calendar clicks now correctly create transactions on selected day
using structured data instead of error-prone string parsing.
